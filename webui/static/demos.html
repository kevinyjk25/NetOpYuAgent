P0-A — Large Payload Cache
ToolResultStore: auto-externalise large tool outputs, retrieve on demand via ref_id
P0runtime/context_budget.pytools/mock_tools.py
📄
Store and page through syslog output (300 lines)
syslog_search returns ~6 KB → stored externally → paginated via ref_id
▶
How it works: When a tool returns > 4,000 chars, ToolResultStore.store() saves the full payload and hands the prompt only a short [STORED:tool:ref_id] Preview: … label. The LLM uses read_stored_result(ref_id, offset) to page through it on demand without re-running the tool or filling the context window.

Hermes §04 — FTS5 Cross-Session Recall
SQLite FTS5 full-text search across all past sessions — retrieved, summarized, injected
Hermes §04memory/fts_store.py
🔍
Write a turn to FTS5 + search past sessions + LLM summary
Simulates new session recall: "what have I seen about RADIUS before?"
▶
The Hermes insight: Instead of dumping full chat history into context (which grows forever), each new session runs a targeted FTS5 keyword search across all past turns. Only the top-K relevant excerpts are retrieved, then LLM-summarized into a compact paragraph before injection. Context stays bounded regardless of how many sessions have run.

Hermes §03 — Active Memory Curation + Nudge
LLM decides what to remember after each turn — not passive storage
Hermes §03memory/curator.py
🗄️
Process 3 turns through the memory curator — see what gets extracted
Per-turn curation + nudge interval tracking
▶
Key difference from passive storage: Traditional systems store every turn. MemoryCurator asks the LLM after each turn: "What from this conversation is worth remembering for the future?" Only distilled facts — device states, operator preferences, incident lessons, tool patterns — are written to long-term memory. The nudge mechanism re-evaluates every 5/20 turns for anything missed.

Hermes §03 — Behavioral User Modeling
Infers operator traits from behavior patterns, not from what they say
Hermes §03memory/user_model.py
👤
Build operator profile from 5 demo turns — domains, tools, traits, contradictions
Stated preferences vs revealed preferences — dialectic reasoning
▶
Hermes Honcho approach: The user model tracks what the operator does, not what they say. If they say "I always test first" but never call a testing tool, that's a contradiction. Technical level is inferred from terminology, communication style from how they respond to different answer lengths. The [OPERATOR PROFILE] is injected as hidden context — the LLM uses it without the operator seeing it.

Hermes §05 — Autonomous Skill Creation
After complex tasks, the agent writes a reusable Skill — agentskills.io format
Hermes §05skills/evolver.py
⚡
Complete RADIUS cert renewal task → auto-create skill
Two-stage: eligibility check (reuse_potential) → LLM writes agentskills.io markdown
▶
Hermes §05: After a complex task completes, the agent asks: "Will this situation come up again?" If reuse_potential ≥ 0.60 and complexity ≥ 3.0, the agent writes a markdown Skill file in agentskills.io format and registers it in the SkillCatalogService. Next time a similar task arrives, the Skill is loaded at Level 2, giving the agent a structured procedure instead of starting from scratch.

Hermes §05 — Skill Self-Improvement via Feedback
Operator feedback patches the Skill's steps — version history + rollback
Hermes §05skills/evolver.py
🔄
Apply feedback to a skill — see version diff + quality delta
LLM patches only the problem step — version chain preserved
▶
Living Skills: A Skill isn't frozen at creation. Every time an operator corrects a step ("Step 2 should check interface metrics first"), the agent re-reads the Skill, generates a targeted patch, and creates SkillVersion(n+1). Quality delta tracks improvement. If a patch makes things worse, rollback to any prior version is one call.
